Segurança da Informação

🔴 Security Headers HTTP Esforço Baixo Impacto Alto Segurança ▼

Cabeçalhos HTTP que instruem o browser a aplicar políticas de segurança — proteção contra XSS, clickjacking, MIME sniffing e vazamento de referrer. Um único middleware resolve tudo.

Por que importa
São gratuitos (zero custo de performance, sem dependências externas), cobrem boa parte do OWASP Top 10 e são verificados em qualquer pentest ou auditoria de segurança contratada pela SEFAZ. Não implementar é deixar vulnerabilidades triviais abertas sem motivo.
Headers a implementar
Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data:
X-Frame-Options: DENY
X-Content-Type-Options: nosniff
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: geolocation=(), microphone=(), camera=()

Um único SecurityHeadersMiddleware registrado globalmente. O Content-Security-Policy exige atenção para não quebrar o Vue/Inertia — precisa ajuste fino nos script-src para assets do Vite.

Bônus de auditoria
Após implementar, testar em securityheaders.com — nota A ou A+ é alcançável e demonstrável para a SEFAZ em qualquer auditoria de segurança. É uma evidência objetiva e visual.
🔴 Audit Log Centralizado Esforço Médio Impacto Muito Alto Segurança ▼

Registro imutável de toda ação relevante no sistema — quem acessou, o que alterou, quando e de onde. Base de qualquer investigação de incidente ou resposta a auditoria.

Por que importa
Sem audit log, um incidente de segurança ou uma disputa contratual com a SEFAZ se torna inauditável. Também é exigido implicitamente pela LGPD (Art. 37 — responsabilização e prestação de contas). É a primeira coisa que qualquer perito forense solicita.
O que registrar
  • Backoffice: login, logout, falhas de autenticação, toda alteração de dados (tenant criado/editado, configuração AR alterada, API key gerada/revogada)
  • API: cada chamada de emissão de boleto — tenant, IP, timestamp, payload resumido
  • Webhooks recebidos: origem, status de validação HMAC, payload resumido
  • Acessos sensíveis: downloads de laudos PDF, visualização de dados de contribuintes
Schema da tabela
audit_logs:
  user_id, tenant_id, action, entity_type, entity_id
  old_values (JSON), new_values (JSON)
  ip_address, user_agent, created_at

-- Arquivos principais:
app/Models/AuditLog.php
app/Http/Middleware/AuditMiddleware.php   ← captura HTTP requests
app/Observers/TenantObserver.php          ← captura eventos Eloquent
app/Services/AuditService.php            ← API para registrar eventos custom
⚠ Requisito crítico — Append-Only
A tabela de audit log deve ser append-only: sem updated_at, sem soft-delete, sem UPDATE possível via ORM. Em produção, o usuário de banco que a aplicação usa não deve ter permissão de DELETE/UPDATE nessa tabela. Se o log puder ser alterado, perde valor jurídico.
🟡 Rate Limiting Granular por Tenant Esforço Baixo Impacto Alto Segurança ▼

Controle de quantas requisições cada tenant pode fazer por endpoint por janela de tempo, usando Redis como contador — infra já disponível no projeto.

Por que importa
Um tenant com bug em produção (loop de emissão) ou mal-intencionado pode derrubar a performance para todos os outros. Com 341k boletos/mês da SEFAZ, um pico inesperado pode comprometer o SLA contratual. Redis já está no stack — é configuração, não nova infraestrutura.
Limites sugeridos por endpoint
EndpointLimiteJanela
POST /api/v1/boletos200 reqpor minuto / tenant
POST /api/v1/boletos/batch10 reqpor minuto / tenant
GET /api/v1/boletos/{id}500 reqpor minuto / tenant
POST /auth/login5 tentativas15 min / IP
GET /api/v1/relatorios10 reqpor minuto / tenant
Headers de resposta informativos

O diferencial é usar a API Key como chave do limitador (não o IP), com headers informativos na resposta — o tenant sabe exatamente onde está dentro do limite:

X-RateLimit-Limit: 200
X-RateLimit-Remaining: 147
X-RateLimit-Reset: 1720900860   ← epoch Unix de quando o limite reseta
Retry-After: 43                 ← segundos até poder tentar novamente (só no 429)

Quando estourar: HTTP 429 com corpo JSON documentado no Swagger, não apenas erro genérico.

🟡 IP Allowlist por Tenant Esforço Baixo Impacto Alto Segurança ▼

Cada tenant cadastra no backoffice os IPs ou faixas CIDR autorizados a chamar a API. Requisições de IPs fora da lista são bloqueadas no middleware antes de chegar ao controller.

Por que importa
Se uma API Key vazar, o atacante ainda precisa vir do IP autorizado — segunda camada de defesa. Para a SEFAZ especificamente, os servidores que emitem boletos têm IPs fixos e conhecidos — é natural e esperado restringir. Campo opcional: sem IPs cadastrados, não aplica restrição (compatibilidade com tenants menores sem IP fixo).
Implementação
-- Campo na tabela tenants:
tenants.allowed_ips (JSON: ["177.xx.xx.xx", "10.0.0.0/24"])

-- Middleware:
app/Http/Middleware/TenantIpAllowlistMiddleware.php

Suporte a CIDR (10.0.0.0/24) além de IPs exatos, para tenants com faixas de IP corporativas. Quando viola: HTTP 403 com registro no audit log informando o IP bloqueado e a API Key utilizada.

🟡 Rotação Automática de API Keys Esforço Médio Impacto Médio Segurança ▼

API Keys com validade configurável (ex: 90, 180 dias ou sem expiração) e notificação automática antes do vencimento, com período de overlap para transição sem downtime.

Por que importa
Credenciais que nunca expiram são uma vulnerabilidade permanente. Se uma key vazou há 6 meses sem ser detectada, o atacante teve acesso por 6 meses. Com rotação, a janela de exposição de qualquer credencial comprometida é limitada ao período de validade.
Fluxo completo de rotação
  1. Admin do tenant gera nova key com validade definida no backoffice
  2. 15 dias antes de expirar: e-mail de aviso via NotificationService
  3. 7 dias antes: segundo aviso com link direto para gerar nova key
  4. No dia da expiração: key desativada automaticamente (scheduler)
  5. Período de overlap (48h): tenant pode ter nova e antiga válidas simultaneamente durante transição, sem interrupção de serviço
-- Campos na tabela api_keys:
api_keys.expires_at       (nullable timestamp)
api_keys.status           (active | expiring_soon | expired | revoked)
api_keys.last_used_at     (timestamp)
api_keys.last_used_ip     (varchar)

-- Jobs (scheduler diário):
app/Jobs/NotifyExpiringApiKeysJob.php
app/Jobs/ExpireApiKeysJob.php
Bônus — Detecção de keys ociosas
Registrar last_used_at + last_used_ip permite identificar keys que não são usadas há muito tempo e desativá-las proativamente — reduz superfície de ataque sem ação manual do tenant.
🟢 Secrets Management (Vault) Esforço Alto Impacto Alto Segurança ▼

Em vez de guardar credenciais sensíveis no .env (PJBank token, Meta access token, senhas ACT ICP-Brasil), elas ficam num cofre criptografado como HashiCorp Vault ou AWS Secrets Manager. A aplicação busca em runtime.

Por que importa
Arquivo .env em produção é copiado em backups, versionado acidentalmente, acessível a qualquer dev com acesso ao servidor. Em uma plataforma com dados fiscais de 341k contribuintes da SEFAZ, isso é inaceitável em qualquer auditoria séria de segurança.
Benefícios concretos
  • Rotação sem redeploy: credenciais trocadas no Vault aplicam imediatamente, sem reiniciar a aplicação
  • Acesso auditado: quem buscou qual secret e quando — trilha completa
  • Secrets por ambiente: dev/staging/prod com acesso segmentado — desenvolvedor nunca vê as credenciais de prod
  • Revogação imediata: credencial comprometida revogada no Vault em segundos, sem alterar código ou fazer deploy
Opções para Payproxy
  • HashiCorp Vault: on-premise ou cloud, open source, integração via AppServiceProvider no boot
  • AWS Secrets Manager: se hospedado na AWS — a instância EC2/ECS tem IAM Role e busca secrets sem credenciais no .env

Credenciais a migrar: PJBank token, Meta WhatsApp access token, ACT ICP-Brasil passwords (BRy, Soluti, Certisign), chaves de criptografia LGPD do CryptoService.

🟢 mTLS nos Webhooks Recebidos Esforço Médio Impacto Médio Segurança ▼

Autenticação mútua TLS (mTLS) — além de validar o HMAC do payload, o servidor Payproxy também valida o certificado TLS de quem está entregando o webhook. Dupla camada de autenticação.

Por que importa
HMAC protege a integridade do payload, mas não autentica o canal de transporte. Um atacante que consiga forjar um payload HMAC válido (por força bruta ou vazamento do secret) ainda seria barrado pelo mTLS. Garante que a conexão TCP/TLS vem de fato da infraestrutura do PJBank.
Como funciona

PJBank apresenta um certificado cliente ao fazer o POST de webhook. Nginx valida esse certificado contra o CA do PJBank antes de encaminhar a requisição ao Laravel. A aplicação não muda — a validação é transparente na camada de transporte:

-- nginx.conf (bloco do servidor Payproxy):
ssl_verify_client on;
ssl_client_certificate /etc/nginx/pjbank-ca.crt;
ssl_verify_depth 2;

Em caso de falha: Nginx retorna 400 No required SSL certificate was sent antes de chegar ao Laravel — zero custo de processamento para requisições ilegítimas.

🟢 Detecção de Anomalias por Tenant Esforço Alto Impacto Alto Segurança ▼

Sistema que aprende o padrão normal de cada tenant (baseline histórico) e alerta quando o comportamento desvia significativamente — volume, horário, concentração de valores.

Por que importa
Um tenant com 50 boletos/dia que emite 2.000 em 10 minutos pode indicar fraude, bug em produção ou comprometimento de credenciais. Sem detecção automática, o dano só é descoberto depois — horas, dias, ou na próxima fatura do PJBank.
Casos de uso concretos
  • Tenant emite 5× mais boletos que a média histórica em 1 hora → alerta por e-mail
  • Tenant com média de 50 boletos/dia emite 2.000 em 10 minutos → bloqueio automático + alerta
  • Concentração anormal — múltiplos boletos com mesmo valor e vencimento → possível fraude em lote
  • Horário incomum — alto volume em madrugada para tenant que nunca emite fora do horário comercial
app/Jobs/ComputeTenantBaselineJob.php   ← noturno; calcula média e desvio padrão
app/Jobs/AnomalyDetectionJob.php        ← a cada 15 minutos
app/Models/TenantBaseline.php           ← histórico: média, desvio, p95 por tenant
Evolução futura
Integração com Slack ou PagerDuty para o canal de on-call. Em v2, tela no backoffice para configurar os thresholds por tenant (ex: SEFAZ com limites mais altos que tenants menores).

LGPD — Lei Geral de Proteção de Dados

🔴 Portal do Titular (Art. 18) Esforço Médio Impacto Muito Alto LGPD ▼

Interface onde o contribuinte (titular dos dados) pode exercer seus direitos previstos no Art. 18 da LGPD: saber quais dados existem sobre ele, receber uma cópia, solicitar correção ou pedir exclusão.

Por que importa — obrigação legal
Sem isso, qualquer reclamação na ANPD de um contribuinte que teve boleto emitido pode resultar em auto de infração. O AR Digital já coleta CPF hash + e-mail — esses dados precisam estar cobertos pelo direito de acesso. Com 341k contribuintes/mês, a probabilidade de uma reclamação formal é real.
Direitos cobertos pelo Art. 18
  • Confirmação de existência: "Vocês têm dados meus?"
  • Acesso: exportar tudo em PDF ou JSON estruturado
  • Correção: solicitar correção de dado incorreto (e-mail, telefone)
  • Anonimização/Exclusão: solicitar remoção — com ressalva explícita sobre retenção fiscal
  • Portabilidade: exportar em formato estruturado para outro controlador
Rotas do portal
GET  /titular/verificar              ← contribuinte informa CPF → recebe link por e-mail
GET  /titular/dados/{token}          ← exibe dados (token de 1 uso / 24h de validade)
POST /titular/exportar/{token}       ← gera PDF com todos os dados do contribuinte
POST /titular/solicitar-exclusao/{token}  ← abre ticket de exclusão/anonimização

O CPF é comparado como hash SHA-256 — já implementado no AR Digital. O contribuinte informa o CPF em texto, o sistema hasheia e busca nos registros.

⚠ Conflito com retenção fiscal
Exclusão de dados de boletos pode conflitar com obrigação de retenção fiscal (5 anos, Art. 195 do CTN). O sistema deve anonimizar (substituir dados pessoais por hash) em vez de deletar, e informar isso explicitamente ao contribuinte no momento da solicitação — evita litígios sobre cumprimento ou não do pedido.
🟡 Retenção e Expurgo Automático Esforço Médio Impacto Alto LGPD ▼

Política definida por tipo de dado com prazo de retenção e job automático que anonimiza ou exclui dados ao atingir o prazo — sem intervenção manual.

Por que importa
A LGPD exige que dados sejam mantidos apenas pelo tempo necessário para a finalidade declarada (princípio da necessidade, Art. 6º). Guardar dados pessoais indefinidamente é infração. Um job automático elimina o risco de acúmulo não intencional ao longo dos anos.

🟡 Consentimento Rastreável para WhatsApp Esforço Baixo Impacto Alto LGPD ▼

Registro com validade jurídica de que o contribuinte autorizou o recebimento de notificações via WhatsApp, conforme exige a Meta e a LGPD para envio de mensagens proativas.

Por que importa — duplo risco
Hoje o WhatsApp é enviado baseado no número de telefone do boleto, sem evidência de que o contribuinte optou por receber via WhatsApp. Isso representa dois riscos simultâneos:
  • Risco Meta: violação dos Termos de Serviço — pode resultar em suspensão do número sem aviso, interrompendo todos os envios do Payproxy
  • Risco LGPD: envio de mensagem proativa sem base legal clara (Art. 7º) — o consentimento é a base mais segura para WhatsApp marketing/notificação
Fluxo de opt-in
  1. Primeiro e-mail de boleto inclui link: "Receber próximos avisos também via WhatsApp? [Sim, autorizo]"
  2. Clique registra: timestamp, IP, versão do texto do consentimento exibido
  3. WhatsApp só é enviado se houver consentimento registrado
  4. Link de opt-out em toda mensagem: "Para não receber mais, clique aqui"
CREATE TABLE whatsapp_consents (
  id, tenant_id, cpf_hash, phone_hash,
  consented_at, consent_text_version,
  revoked_at, revocation_ip
);
🟡 Data Masking nas Interfaces Esforço Médio Impacto Médio LGPD ▼

Dados pessoais exibidos nas telas do backoffice são automaticamente mascarados baseando-se no perfil do usuário logado — acesso completo apenas para quem tem permissão explícita.

Por que importa
Minimização de exposição — um atendente de suporte vê o suficiente para resolver o problema sem acesso ao dado pessoal completo. Princípio de mínimo privilégio aplicado na camada de apresentação. CryptoService já existe no projeto — é questão de UI e API Resource.

🟡 Workflow de Notificação de Incidente (Art. 48) Esforço Médio Impacto Alto LGPD ▼

Processo estruturado para detecção, avaliação e notificação de incidentes de segurança envolvendo dados pessoais, respeitando os prazos legais da LGPD (72h para notificação à ANPD).

Por que importa
A LGPD exige notificação à ANPD em até 72 horas após a detecção do incidente, e notificação aos titulares afetados em prazo razoável. Sem processo estruturado, o prazo é perdido na confusão do momento — o resultado é multa garantida além do dano de imagem.
Fluxo automatizado (8 etapas)
  1. Detecção — manual ou automática via Detecção de Anomalias
  2. Abertura do incidente no sistema (app/Models/SecurityIncident)
  3. Avaliação automática de risco — quantos titulares afetados, quais dados expostos, gravidade
  4. Notificação interna imediata — e-mail para DPO + administrador Payproxy
  5. Timer de 72h visível no backoffice com countdown regressivo até o prazo ANPD
  6. Geração do formulário pré-preenchido para notificação à ANPD com todos os dados disponíveis
  7. Registro de todas as ações tomadas — trilha de evidências auditável
  8. Notificação aos titulares afetados via e-mail com template aprovado juridicamente
app/Models/SecurityIncident.php
app/Services/IncidentNotificationService.php
O que o sistema gera automaticamente
Rascunho do comunicado à ANPD com: dados do incidente, sistemas afetados, número estimado de titulares impactados, medidas tomadas imediatamente. O DPO revisa e envia — mas a base documental já está pronta, evitando erros sob pressão de prazo.
🟢 Registro de Operações de Tratamento (Art. 37) Esforço Alto Impacto Alto LGPD ▼

Inventário documentado de todas as operações de tratamento de dados pessoais realizadas pela plataforma — exigido pelo Art. 37 da LGPD para controladores e operadores que tratam dados em larga escala.

Por que importa
Para a SEFAZ (341k contribuintes/mês), a ANPD pode solicitar esse inventário a qualquer momento, especialmente em uma renovação de contrato ou investigação. Mantê-lo no sistema — e não em planilha — garante que está sempre atualizado conforme novos módulos são implementados.

🟢 RIPD Automatizado Esforço Alto Impacto Médio LGPD ▼

Relatório de Impacto à Proteção de Dados Pessoais (RIPD) — documento exigido pela ANPD para operações de tratamento de alto risco. Com 341k contribuintes/mês tratando CPF e dados fiscais, Payproxy se enquadra nessa categoria.

Por que importa
O RIPD é o documento mais completo de demonstração de maturidade em privacidade. É diferencial competitivo concreto na renovação do contrato com a SEFAZ e em auditorias da ANPD — poucos SaaS brasileiros têm isso documentado e atualizado formalmente.
O que o RIPD documenta
  • Descrição dos processos de tratamento — como e onde os dados fluem no sistema
  • Necessidade e proporcionalidade — justificativa para cada dado coletado
  • Riscos identificados para os titulares — vazamento, uso indevido, discriminação
  • Medidas de mitigação adotadas — controles técnicos e organizacionais implementados
  • Avaliação residual de risco — parte preenchida manualmente pelo DPO
Como o sistema gera automaticamente

Cruza os dados do Registro de Operações de Tratamento com as medidas de segurança implementadas (catalogadas no sistema) e gera um PDF nos moldes do formulário da ANPD. O DPO preenche apenas a avaliação de risco subjetiva — o sistema monta todo o restante do documento final, pronto para entrega.
