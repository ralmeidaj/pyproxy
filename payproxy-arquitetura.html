2. Arquitetura da Aplicação

Padrão Service Layer com Strategy Pattern para parceiros bancários. Controllers thin — apenas orquestram. DTOs eliminam dependência do objeto Request dentro dos Services.

Fluxo: Request → Middleware → FormRequest → Controller → DTO → Service → BankPartnerInterface → PJBankService → API Resource → Response

Services Principais

ServiceResponsabilidade
BoletoServiceEmissão, cancelamento e consulta de boletos via PJBank
SplitServiceCálculo e validação de splits de pagamento
NotificationServiceOrquestra e-mail + WhatsApp por modelo de mensageria
WebhookDeliveryServiceEntrega de webhooks para tenants com retry e HMAC
ReconciliationServiceConciliação de pagamentos PJBank vs banco local
ReportServiceGeração de relatórios XLSX e CSV
CryptoServiceCriptografia de dados sensíveis (LGPD)
SanitizationServiceSanitização de CPF, CNPJ, e-mail, telefone
ArDigitalServiceOrquestra o fluxo AR Digital completo
ArTrackingServicePixel, token, hash SHA-256, confirmação CPF
ArEvidencePdfServiceGera laudo PDF (DomPDF + QR code RFC 3161)
Rfc3161TimestampServiceCarimbos de tempo RFC 3161 via ACT ICP-Brasil
BankPartnerFactoryFactory do Strategy Pattern — retorna adapter correto
PJBankServiceAdapter PJBank — único parceiro ativo em v1

7. Módulo AR Digital

Add-on por tenant que fornece prova jurídica de entrega e recebimento de boletos, com cadeia de evidências RFC 3161 ICP-Brasil e laudo PDF.

Tabelas

TabelaPropósito
ar_digital_configsConfiguração por tenant — enabled, pixel_tracking, cpf_confirmation, act_provider
ar_digital_notificationsUma por boleto com AR ativo — token UUID, status, hash do documento
ar_digital_eventsCada evento rastreado (envio, entrega, abertura, confirmação, bounce)
ar_digital_timestampsCarimbo RFC 3161 por evento — TSR base64, provedor ACT, hash_input

Progressão de Status

enviado → entregue → lido → confirmado  (progressão normal)
enviado → bounce  (falha de entrega — terminal)

Eventos e Carimbos RFC 3161

#EventoGatilhoConfigurável
1envioE-mail despachado pelo servidorNão — sempre
2entrega_provedorDSN SMTP 2xx (servidor aceitou)Não — sempre
3aberturaPixel 1×1 carregado no clienteSim — pixel_tracking
4confirmacao_cpfContribuinte confirma CPF na landingSim — cpf_confirmation
